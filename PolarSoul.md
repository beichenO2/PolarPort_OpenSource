# PolarPort — PolarSoul

## 设计哲学

PolarPort 是 Polarisor 的端口分配服务，为所有项目提供幂等、可靠、有规则的端口管理。

- **幂等分配**: 同一服务反复请求端口，永远拿到同一个——消除配置漂移
- **心跳保活**: 30 秒无心跳即标记 stale，过期端口可被新服务复用
- **端口合规**: preferred_port 必须以 0 或 5 结尾，便于记忆和规避常见端口冲突

## 功能介绍

- **生态位**: 端口注册表，所有服务的端口由 PolarPort 统一分配和追踪
- **承担功能**:

| 编号 | 功能域 | 说明 |
|---|---|---|
| R1 | 端口分配 | `POST /api/allocate`——幂等分配，同一 service_name + project 始终返回同一端口 |
| R2 | 端口释放 | `POST /api/release`——手动释放端口 |
| R3 | 心跳保活 | `POST /api/heartbeat`——30 秒窗口，超时标记 stale |
| R4 | 端口查询 | `GET /api/list`——列出所有已分配端口及其状态 |

## 与其他项目的关系

- **从 SOTAgent 分拆而来**: 端口管理原为 SOTAgent R1 的一部分，因端口逻辑独立性强而分拆
- **被所有服务依赖**: 每个需要监听端口的 Polarisor 服务都通过 PolarPort 获取端口
- **SOTAgent 是特殊消费者**: SOTAgent 自身也通过 PolarPort 获取端口（自举）

## 关键设计决策

### Why 幂等分配

**问题**: 服务重启后分配到不同端口，导致其他服务的硬编码端口引用失效。

**决策**: 同一 service_name + project 始终返回同一端口，消除重启时的端口漂移。

**不可妥协**: 端口分配必须幂等，这是整个生态服务发现的基础。

### Why 端口以 0/5 结尾

**问题**: 随机端口难以记忆，且可能与系统服务冲突。

**决策**: 强制端口以 0 或 5 结尾，形成可预测的端口分配模式。
